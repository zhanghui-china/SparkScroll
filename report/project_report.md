# SparkScroll项目报告书

## 1. 项目概述

### 1.1 项目名称
SparkScroll - 智能多模态内容生成与编辑平台，中文名称为”星火绘卷“。 Spark 直接致敬Nvidia DGX Spark平台，Scroll代表连环画、画卷。中文“星火绘卷”既有科技感，又有古典文学韵味。

### 1.2 项目目标
‍开发一个基于 NVIDIA GPU 加速的智能多模态内容生成与编辑平台，集成先进的大语言模型和图像编辑模型，为用户提供高效、高质量的内容创作工具。

该平台能够输入故事长文本，通过多 Agent 协作系统，全自动进行剧情压缩、分镜拆解、角色立绘设定、高保真图像渲染与后期排版，最终流式输出具有高度角色一致性的多页连环画。

### 1.3 项目背景
随着AI技术的快速发展，多模态内容生成已经成为内容创作领域的重要趋势。然而，现有的内容生成工具往往存在以下问题：
- 单模态功能限制，无法满足多场景需求
- 计算资源需求高，普通设备难以流畅运行
- 模型部署复杂，用户使用门槛高
- 生成质量参差不齐，缺乏专业性

SparkScroll 项目正是为解决这些痛点而诞生，依托 NVIDIA DGX Spark 平台的强大算力，实现高效、高质量的多模态内容生成。

### 1.4 项目要解决的实际问题
1. 降低AI模型使用门槛，让普通用户也能轻松使用先进的AI模型
2. 提供一站式多模态内容生成解决方案，涵盖文本和图像编辑
3. 利用NVIDIA GPU加速技术，提高模型推理速度和效率
4. 确保生成内容的质量和专业性，满足商业级应用需求

## 2. 作品描述

### 2.1 作品亮点

**2.1.1 平台专属优化，性能极致领先** 

本项目专为 NVIDIA DGX Spark (GB10 128G 统一内存)量身定制，采用“重型双大模型常驻显存”架构，总显存占用约 115GB，普通单卡系统完全无法运行该并发管线。只有NVIDIA DGX Spark 能实现“零 Swap”的极致秒级响应，支撑多 Agent 极速切换与流式呈现，彻底拉开与普通消费级显卡的差距。

**2.1.2 高度还原工业化出版流程** 

摒弃了简单的端到端黑盒，设计了高度还原人类工业化出版流程的 6 阶段 Agent 工作流，包括导演 Agent、编剧 Agent、立绘 Agent、出稿 Agent、后期引擎和流式呈现，实现从文本输入到最终连环画输出的全自动化流程。

**2.1.3 角色一致性与内容质量保障** 

通过立绘 Agent 预先生成角色参考图，解决了 AI 绘图“角色不连贯”的难题；采用纯代码进行后期配字，拒绝 AI 生字乱码，确保内容质量。

**2.1.4 多模态集成及良好的可扩展性** 

集成了Qwen3.5-9B等大语言模型和Qwen-Image-Edit-2511、FireRed-Image-Edit-1.1等图像编辑模型，实现文本和图像的无缝生成与编辑，提供本地模型和云端模型的任意搭配，模块化设计，支持后续集成更多先进模型。

**2.1.5 模型本地部署**

支持模型本地部署，保证数据隐私和离线使用能力。

### 2.2 作品功能

**2.2.1 核心功能** 

输入故事（如世界经典名著）长文本，通过多 Agent 协作系统，全自动进行剧情压缩、分镜拆解、角色立绘设定、高保真图像渲染与后期排版，最终流式输出具有高度角色一致性的多页连环画。

**2.2.2 功能模块说明** 

| 功能模块       | 具体描述                                                     |
| -------------- | ------------------------------------------------------------ |
| **导演 Agent** | 对原始长文本进行“降维压缩”，剔除冗余支线，精简核心角色与关键场景，提炼出紧凑的主线剧本。 |
| **编剧 Agent** | 将导演的剧本进行视觉化“镜头拆分”，规划每页 4-6 个场景，整部连环画篇幅控制在 20 页以内，输出标准化 JSON 剧本（含场景描述、留白坐标预估、对白）。 |
| **立绘 Agent** | 在正式绘制分镜前，根据导演设定的角色名单，预先绘制“核心角色三视图”与“关键物件图”，为后续分镜提供风格对齐基准。 |
| **出稿 Agent** | 接收编剧的 JSON 分镜 Prompt 与立绘 Agent 的角色参考图，生成图像时预留“构图留白”，合并输出每页的基础图像。 |
| **后期引擎**   | 通过纯代码读取编剧 JSON 中的对白文本与坐标，加载中文字体模板，将文字完美渲染至出稿 Agent 预留的空白区域。 |
| **流式呈现**   | 前端采用“流式按页呈现”策略，每生成完一页即刻推送到前端供用户翻阅，理想状态下可生成连贯的 PDF 画册。 |

**2.2.3 功能规划原则 (Planning Principles)** 

1. **先打通主链路**：优先保证“文本接入 -> 导演 -> 编剧 -> 页面产出 -> 预览”闭环可运行。
2. **先保证一致性，再扩能力**：优先解决角色一致性、页面可读性、状态可恢复。
3. **严格沿用既有术语**：统一使用 `project_id`、`episode`、`page`、`/api/v1`、`CREATED/READY/RUNNING/...` 等既有命名。
4. **按资源约束设计**：遵循文本模型 1 并发、图像模型 1 并发、跨模态可并行的基础约束。

## 3. 技术架构

### 3.1 技术创新点

**3.1.1 重型双大模型常驻显存架构** 

采用“文本与逻辑主脑（Qwen3.5-9B）+ 视觉渲染中枢（Qwen-Image-Edit-2511）”的双大模型架构，总显存占用约 115GB，实现几万字名著“一口气”读入并保持全局剧情连贯，以及极高保真度的参考图重绘与构图留白。

**3.1.2 多 Agent 协作流水线** 

设计 6 阶段 Agent 工作流，高度还原人类工业化出版流程，每个 Agent 专注于特定任务，通过标准化 JSON 协议流转，实现高效协作。

**3.1.3 角色一致性解决方案** 

通过立绘 Agent 预先生成角色参考图，后续所有分镜均以此为基准进行风格对齐，解决了 AI 绘图中角色不连贯的行业难题。

**3.1.4 程序化后期配字** 

采用 Python/PIL 图像处理库进行纯代码配字，拒绝 AI 生字乱码，确保文字渲染的准确性和美观性。

**3.1.5 流式呈现技术** 

前端采用流式按页呈现策略，极大优化用户体验，无需等待全部内容生成即可开始翻阅。

**3.1.6 GPU优化策略**

采用内存优化技术，通过调整`gpu-memory-utilization`参数，在保证性能的同时降低内存使用，使模型能够在有限资源环境下运行。

**3.1.7 API标准化**

 设计了标准化的API接口，支持不同模型的统一调用方式，提高系统的可扩展性和可维护性。允许用户根据自己的业务定义UI界面与API服务端对接。



### 3.2 平台适配性与不可替代性 (Platform Exclusivity)

本项目专为 **NVIDIA DGX Spark (GB10 128G 共享显存)** 量身定制。为彻底拉开与普通消费级显卡（如 RTX 4090 24G）的差距，SparkScroll 采用了 **“重型双大模型常驻显存” (Memory-Resident Multi-Model)** 架构：

| 模块               | 选型                          | 显存常驻预算      | GB10 的绝对必要性                                            |
| :----------------- | :---------------------------- | :---------------- | :----------------------------------------------------------- |
| **文本与逻辑主脑** | **Qwen3.5-9B**                | **~40GB**         | 需开启 262k 超长上下文 (KV Cache 满载) 及 Thinking 模式，以实现几万字名著“一口气”读入并保持全局剧情连贯。 |
| **视觉渲染中枢**   | **Qwen-Image-Edit-2511**      | **~60GB**         | 作为基于 MMDiT 架构的 57GB 级巨型图像编辑模型，负责极高保真度的参考图重绘与构图留白。 |
| **框架与并发缓冲** | TensorRT-LLM / Diffusers / OS | **~15GB**         | 支撑 API 高速流转、多模态张量转换与 Batch 并发出图。         |
| **总计**           | --                            | **~115GB / 128G** | **普通单卡系统完全无法运行该并发管线。** 只有 GB10 能实现“零 Swap”的极致秒级响应，支撑多 Agent 极速切换与流式呈现。 |

### 3.4 整体分层架构设计

系统采用分层架构，确保“接口、调度、资源、状态、模型、安全、观测”彼此解耦：

1. **表现层 (API Layer)**  
   基于 **FastAPI** 提供 RESTful 接口，负责项目创建、参数校验、文本接入、语义保护切分、任务启动/停止、进度查询、页面预览和提示词管理。

2. **调度层 (Orchestration Layer)**  
   基于 **LangChain Graph / LangGraph** 组织多 Agent 工作流，负责节点编排、执行状态流转、失败重试、同步点控制和队列投递。

3. **资源管理层 (Resource Management Layer)**  
   统一管理本地文件系统资源，负责系统级公共预设、项目级私有资源、生成图片资产及资源隔离校验。

4. **运行态数据与记忆层 (Runtime State & Memory Layer)**  
   基于 **Redis** 管理项目元数据、队列、锁、运行状态、单集内容、分镜结果、导演/编剧记忆、停止信号等高频动态数据。

5. **模型适配层 (Model Gateway)**  
   基于 **TensorRT-LLM**、本地推理服务和兼容云端 API 的适配器层，负责逻辑模型与视觉模型的统一调用、超时控制、错误重试与本地/云端切换。

6. **安全执行层 (Security & Sandbox Layer)**  
   基于 **NVIDIA OpenShell** 提供 Agent 运行沙箱，确保文件访问、网络访问和密钥使用均受 YAML 策略约束。

7. **观测与诊断层 (Observability Layer)**  
   基于 **loguru** 与结构化日志规范，覆盖系统运行日志、项目运行日志、API 审计日志和问题定位链路。

   

### 3.5 资源管理层设计

资源管理层负责管理**本地文件系统中的低频、可持久化、可复用资产**，并严格执行项目边界控制。

#### 3.5.1 访问隔离原则

*   **严格按 Project ID 隔离**：除系统级公共预设外，所有项目资产均必须在对应项目 ID 命名空间下读写。
*   **禁止跨项目读取资源信息**：任何 Agent 不允许读取其他项目的原始文本、角色资产、分镜文件、底图和页面文件。
*   **系统公共预设只读访问**：通用的连环画构图框架、排版样式等系统预设可被所有项目读取，但必须保持只读，不允许被项目运行过程覆盖。
*   **停止任务不删除已产物**：调用停止接口只中断后续执行，不删除已经落盘的文本切片、角色资源、分镜结果、底图和页面资源。

#### 3.5.2 资源分类

1. **系统级公共预设**  
   包含通用项目预设的连环画构图框架、排版样式、页面布局规则、文字框位置规则等公共模板资产。此类资源由系统维护，全项目共享，只读访问。

2. **项目文本资源**  
   包含原始小说文本及经 API 层语义保护切分后的 `.txt` / `.md` 片段。切分结果是导演 Agent 的直接输入来源。

3. **项目模版资源**  
   包含该项目使用的构图模版、页面排版模版、镜头样式规则和页面渲染配置，主要以 `.json` 为核心描述格式。

4. **角色资源库**  
   仅包含两类资产：

   *   **角色图片**：`.jpg`、`.png`
   *   **角色配置**：`.json`

   角色资源库不存放 Markdown 文稿或额外文本说明，角色语义信息统一通过 JSON 配置承载，角色视觉参考统一通过图片承载。

5. **图像与页面资产**  
   包含立绘结果图、出稿底图、中间编辑图、最终渲染页面及页面聚合清单等资产。



### 3.6 运行态数据与 Redis 管理设计

Redis 负责承载**高频读写、任务调度、状态追踪和运行态记忆**，是系统的动态控制中枢。

| 数据类别   | 推荐 Key 示例                           | 说明                                                         |
| :--------- | :-------------------------------------- | :----------------------------------------------------------- |
| 项目元数据 | `project:{id}:meta`                     | 存储项目名称、模型选择、创建时间、分片摘要等基础信息。       |
| 项目状态   | `project:{id}:status`                   | 存储项目状态机，如 `CREATED / READY / RUNNING / STOPPED / COMPLETED / FAILED`。 |
| 集列表     | `project:{id}:episodes`                 | 存储项目下已规划集的顺序索引。                               |
| 单集内容   | `project:{id}:episode:{ep}:outline`     | 存储导演生成的单集纲要、角色新增摘要、剧情目标。             |
| 分镜内容   | `project:{id}:episode:{ep}:storyboards` | 存储编剧输出的镜头 JSON 列表。                               |
| 导演记忆   | `project:{id}:memory:plot`              | 存储宏观剧情梗概、人物关系、长期伏笔。                       |
| 编剧记忆   | `project:{id}:memory:scene`             | 存储最近若干镜头状态，确保镜头衔接一致。                     |
| 任务队列   | `queue:text` / `queue:image`            | 文本任务与图像任务分队列排队执行。                           |
| 资源锁     | `lock:model:text` / `lock:model:image`  | 控制文本模型与图像模型各自仅 1 并发。                        |
| 停止信号   | `project:{id}:control:stop`             | 标记项目已发出停止请求，Worker 轮询并安全退出。              |
| 任务运行态 | `project:{id}:task:{task_id}`           | 存储单任务状态、重试次数、耗时、错误信息。                   |

Redis 中保存的是**运行态数据**而非长期冷存档；长期可复用的页面与图像结果应落盘到资源管理层，避免 Redis 承担大体积二进制资源存储。



### 3.7 核心数据对象与状态机 (Domain Model & State Machine)

#### 3.7.1 核心数据对象

系统建议统一以下核心对象模型：

1.  **Project**  
    表示一次完整的连环画生成任务，包含模型配置、切片信息、资源根标识、状态机和统计信息。
2.  **Episode**  
    表示导演规划出的单集单元，是导演、编剧、立绘、出稿协同的最小业务单元之一。每一集对应一份纲要、一组角色增量和一组分镜结果。
3.  **Scene / Shot**  
    表示编剧输出的单个镜头，是后续生图、排版和页面合成的直接依据。推荐以 JSON 对象形式描述构图、人物、动作、对白和镜头说明。
4.  **Character Asset**  
    表示角色资源库中的单角色资产集合，包括角色主图、补充参考图和角色配置 JSON。
5.  **Page**  
    表示最终输出的单页连环画结果，包含页面编号、底图、渲染图、文本覆盖状态和预览地址。

#### 3.7.2 项目状态机

项目状态机按如下约束实现：

| 状态        | 含义                                         | 允许的下一状态                      |
| :---------- | :------------------------------------------- | :---------------------------------- |
| `CREATED`   | 项目已创建，但尚未完成切片初始化             | `READY` / `FAILED`                  |
| `READY`     | 切片与资源目录已就绪，可启动                 | `RUNNING` / `FAILED`                |
| `RUNNING`   | 正在执行多 Agent 流水线                      | `STOPPING` / `COMPLETED` / `FAILED` |
| `STOPPING`  | 已接收停止指令，等待 Worker 安全退出         | `STOPPED` / `FAILED`                |
| `STOPPED`   | 已停止，但已完成产物保留，可重新启动继续执行 | `RUNNING` / `FAILED`                |
| `COMPLETED` | 全流程完成                                   | --                                  |
| `FAILED`    | 出现不可恢复错误                             |                                     |

#### 3.7.3 任务状态机

所有 Agent 任务统一复用以下任务状态：
`PENDING -> QUEUED -> RUNNING -> SUCCEEDED / FAILED / CANCELLED`

其中：

*   `CANCELLED` 表示任务因停止信号或人工中止而退出。
*   已经完成的任务不会因为项目进入 `STOPPED` 而被回滚。
*   `start` 在 `STOPPED` 状态下应表现为**恢复执行**，而不是重新清空已有内容。



### 3.8 核心多 Agent 协作流水线 (Pipeline Architecture)

SparkScroll 摒弃了简单的端到端黑盒，设计了高度还原人类工业化出版流程的 **6 阶段 Agent 工作流**：

#### 3.8.1 原始输入与分段 (User Input)

* **动作**：用户输入名著或小说原文原文。
* **处理**：支持通过行号或正则截断表达式进行初步的章节分段。

#### 3.8.2 进度管理中枢 (Monitor System)

* **动作**：贯穿全生命周期的系统级调度器。
* **处理**：实时向用户前端汇报工作流进展（如：“导演已完成文本压缩”、“编剧正在规划第3页，当前场景 12/120”），提供清晰的进度感知。

#### 3.8.3 导演 Agent (Director - 文本驱动)

* **动作**：Qwen3.5-9B (开启长上下文阅读)。
* **处理**：对原始长文本进行“降维压缩”。剔除冗余支线，精简核心角色与关键场景，提炼出紧凑的主线剧本。

#### 3.8.4 编剧 Agent (Writer - 文本驱动)

* **动作**：Qwen3.5-9B (模板化结构输出)。
* **处理**：将导演的剧本进行视觉化“镜头拆分”。
* **规则限制**：严格通过 Prompt 模板控制节奏，规划**每页 4-6 个场景**。整部连环画篇幅控制在 20 页以内，总计不超过 120 个场景。输出标准化 JSON 剧本（含场景描述、留白坐标预估、对白）。

#### 3.8.5 立绘 Agent (Prop/Character Designer - 图像驱动)

* **引擎**：Qwen-Image-Edit-2511 (Diffusers 驱动)。
* **任务**：在正式绘制分镜前，根据导演设定的角色名单，预先绘制“核心角色三视图”与“关键物件图”（Reference Sheets）。
* **意义**：这是解决 AI 绘图“角色不连贯”的杀手锏。后续所有分镜均以此为基准进行风格对齐。

#### 3.8.6 出稿 Agent (Drafting & Painting - 图像驱动)

* **引擎**：Qwen-Image-Edit-2511 (Diffusers 混合生图 API)。
* **任务**：接收编剧的 JSON 分镜 Prompt 与立绘 Agent 的角色参考图。在生成图像时，根据 JSON 预留“构图留白”（供后续填字）。合并输出每页的基础图像。

#### 3.8.7 后期引擎 (Editor System - 程序化驱动)

* **引擎**：Python / PIL (Pillow) 图像处理库。
* **任务**：**拒绝 AI 生字乱码**。通过纯代码读取编剧 JSON 中的对白文本与坐标，加载中文字体（如黑体/楷体）模板，将文字完美渲染至出稿 Agent 预留的空白区域（对话框或旁白栏）。

#### 3.8.8 组装与流式呈现 (Assembler & Viewer)

* **机制**：前端采用“流式按页呈现”策略。不需要等待全部 120 个场景跑完，**每生成完一页（4-6 镜 + 后期配字），即刻推送到前端供用户翻阅**，极大优化黑客松演示体验。
* **扩展**：理想状态下，最终生成连贯的 PDF 画册（初期版本设为 Optional 目标）。

## 4. 使用的NVIDIA工具平台SDK开源模型

**4.1 平台适配** 

本项目专为 NVIDIA DGX Spark (GB10 128G 共享显存)平台量身定制，充分利用平台的大显存优势，实现双大模型常驻显存运行。

**4.2 工具与模型使用**

- **NVIDIA CUDA**：使用CUDA 12.9/13.0作为GPU加速基础，充分利用NVIDIA GPU的并行计算能力
- **PyTorch**：使用PyTorch 2.10.0作为深度学习框架，结合CUDA实现高效的模型推理
- **TensorRT-LLM** ：部署 Qwen3.5-9B 大语言模型，优化推理延迟，实现 128k 超长上下文处理。
- **VLLM**：使用VLLM作为本地模型的推理引擎，通过张量并行和内存优化技术，大幅提升模型推理速度
- **HuggingFace Diffusers** ：原生驱动 Qwen-Image-Edit-2511 图像编辑模型，灵活注入 Image/Prompt 条件。
- **NVIDIA GB10 硬件** ：在NVIDIA DGX Spark平台上进行模型测试和部署，提供 128G 共享显存，支撑双大模型常驻显存运行，实现“零 Swap”的极致秒级响应。
