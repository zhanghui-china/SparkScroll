# SparkScroll项目报告书

## 1. 项目概述

### 1.1 项目名称
SparkScroll - 智能多模态内容生成与编辑平台，中文名称为"星火绘卷"。 Spark 直接致敬Nvidia DGX Spark平台，Scroll代表连环画、画卷。中文“星火绘卷”既有科技感，又有古典文学韵味。

### 1.2 项目目标
‍开发一个基于 NVIDIA GPU 加速的智能多模态内容生成与编辑平台，以 gpt-5.4 的规划能力和自研 Agent 调用逻辑、调度实现为核心，整合先进的大语言模型和图像编辑模型，为用户提供高效、高质量的内容创作工具。

该平台能够输入故事长文本，通过多 Agent 协作系统，全自动进行剧情压缩、分镜拆解、角色立绘设定、高保真图像渲染与后期排版，最终流式输出具有高度角色一致性的多页连环画。

### 1.3 项目背景
在全球人工智能技术从单一维度的文本对话向多模态、多智能体（Multi-Agent）深度协同演进的关键历史节点，数字内容创作产业正处于一场前所未有的生产力重构之中。长久以来，传统的连环画、漫画与图像小说出版行业受制于极其高昂的人力成本、漫长的制作周期以及难以逾越的专业绘画技能壁垒 。随着AI技术的快速发展，**多模态内容生成**已经成为内容创作领域的重要趋势。

然而，现有的内容生成工具往往存在以下问题：

- 单模态功能限制，无法满足多场景需求
- 计算资源需求高，普通设备难以流畅运行
- 模型部署复杂，用户使用门槛高
- 生成质量参差不齐，缺乏专业性

SparkScroll 项目正是为解决这些痛点而诞生，依托 NVIDIA DGX Spark 平台的强大算力，实现高效、高质量的多模态内容生成。

### 1.4 项目要解决的实际问题
“算力破局，AI 落地”是本次 NVIDIA DGX Spark 全栈 AI 开发黑客松的根本立意，SparkScroll 在实际生产生活中所展现出的广阔用途，则是赋予该项目灵魂的血肉。它远不止于一个“自动画漫画”的工具，更是一场重塑文化传承、教育普及和内容商业模式的底层革命。

1. 降低AI模型使用门槛，让普通用户也能轻松使用先进的AI模型
2. 提供一站式多模态内容生成解决方案，涵盖文本和图像编辑
3. 利用NVIDIA GPU加速技术，提高模型推理速度和效率
4. 确保生成内容的质量和专业性，满足商业级应用需求

## 2. 作品描述

### 2.1 作品亮点

**2.1.1 DGX Spark平台专属优化，性能极致领先** 

传统的消费级算力平台（例如搭载 24GB 显存的顶级显卡 RTX 4090，或通过 PCIe 总线桥接的多卡环境）在处理多模态、长上下文的工业级多智能体流水分发时，面临着难以逾越的“内存墙”与“数据传输瓶颈” 。在传统架构中，当多个大语言模型与扩散模型需要交替工作时，显存容量的不足迫使系统必须频繁地在显存（VRAM）与系统内存（RAM）之间进行模型权重的卸载与加载（Swap）。这种跨越 PCIe 总线的数据搬运会引发断崖式的延迟，在处理超长上下文（如 100K 以上）时，提示词处理速度的衰减甚至可达 64 倍之多。

DGX Spark 搭载的 GB10（Grace Blackwell）超级芯片从物理层面彻底消除了这一瓶颈。GB10 采用了创新的 SoC 设计，通过 NVLink-C2C 技术将 Grace CPU 与 Blackwell GPU 紧密集成，实现了高达 900 GB/s 的双向芯片间通信带宽 。更为关键的是，该平台提供了 128GB 的 LPDDR5x 统一内存，内存带宽达到 273 GB/s，使得 CPU 与 GPU 共享同一物理内存寻址空间。这意味着推理引擎在处理巨量文本的 KV 缓存（KV Cache）以及庞大的扩散模型隐空间（Latent Space）张量时，实现了真正意义上的“零数据拷贝”，从而支撑起多智能体之间的极速轮转交互。

SparkScroll 项目专为 NVIDIA DGX Spark (GB10 128G 统一内存)量身定制，采用"重型双大模型常驻显存"架构，总显存占用约 115GB，普通单卡系统完全无法运行该并发管线。只有NVIDIA DGX Spark 能实现**"零 Swap"**的极致响应，支撑多 Agent 极速切换与流式呈现，**彻底拉开与普通消费级显卡的差距**。

**2.1.2 高度还原工业化出版流程** 

摒弃了简单的端到端黑盒，基于自研六阶段 Agent runtime 还原人类工业化出版流程，涵盖导演（Director Agent）、编剧 （Writer Agent）、立绘（CharacterDesigner Agent）、出稿（Drafting Agent）、剪辑（Editor Agent）、合成（Assembler Agent） 六个阶段。其中 Editor / Assembler 已在 Gateway 中接入 runtime，但当前 1.0 仍以 pass-through loop 形式运行。

**2.1.3 角色一致性与内容质量保障** 

通过立绘 Agent 预先生成角色参考图，解决了 AI 绘图“角色不连贯”的难题；当前版本的文字渲染主要由图片模型直接完成，其中 FireRed-Image-Edit-1.1 在中文字幕渲染上表现突出。后续可采用纯代码进行后期配字，拒绝 AI 生字乱码，确保内容质量。

**2.1.4 多模态集成及良好的可扩展性** 

文本侧采用 Qwen3.5-9B，图像侧支持 Qwen-Image-Edit-2511 与 FireRed-Image-Edit-1.1 两种本地模型；驱动层同时支持 Diffusers（稳定）与 vLLM-Omni（速度快），并通过统一 provider 契约便于切换本地与云端模型。

**2.1.5 模型本地部署**

支持模型本地部署，保证数据隐私和离线使用能力。

### 2.2 作品功能

#### 2.2.1 核心功能

输入故事（如世界经典名著）长文本，通过多 Agent 协作系统自动完成剧情压缩、分镜拆解、角色立绘设定与页面生成，最终按页流式输出具有高度角色一致性的多页连环画。独立 Editor System 当前仍处于开发中，现阶段主展示链路主要使用图片模型完成文字渲染。

#### 2.2.2 功能模块说明

| 功能模块            | 具体描述                                                     |
| ------------------- | ------------------------------------------------------------ |
| **导演 Agent**      | 对原始长文本进行“降维压缩”，剔除冗余支线，精简核心角色与关键场景，提炼出紧凑的主线剧本。 |
| **编剧 Agent**      | 将导演的剧本进行视觉化“镜头拆分”，规划每页 4-6 个场景，整部连环画篇幅控制在 20 页以内，输出标准化 JSON 剧本（含场景描述、留白坐标预估、对白）。 |
| **立绘 Agent**      | 在正式绘制分镜前，根据导演设定的角色名单，预先绘制“核心角色三视图”与“关键物件图”，为后续分镜提供风格对齐基准。 |
| **出稿 Agent**      | 接收编剧的 JSON 分镜 Prompt 与立绘 Agent 的角色参考图，生成图像时预留“构图留白”，合并输出每页的基础图像。 |
| **剪辑 &合成Agent** | 尚在开发中，暂未在当前主链路启用。现阶段主要由图片模型直接完成中文文字渲染，后期将通过纯代码读取编剧 JSON 中的对白文本与坐标，加载中文字体模板，将文字完美渲染至出稿 Agent 预留的空白区域。 |
| **流式呈现**        | 前端采用“流式按页呈现”策略，每生成完一页即刻推送到前端供用户翻阅，理想状态下可生成连贯的 PDF 画册。 |

#### 2.2.3 功能规划原则 (Planning Principles)

1. **先打通主链路**：优先保证“文本接入 -> 导演 -> 编剧 -> 页面产出 -> 预览”闭环可运行。
2. **先保证一致性，再扩能力**：优先解决角色一致性、页面可读性、状态可恢复。
3. **严格沿用既有术语**：统一使用 `project_id`、`episode`、`page`、`/api/v1`、`CREATED/READY/RUNNING/...` 等既有命名。
4. **按资源约束设计**：遵循文本模型 1 并发、图像模型 1 并发、跨模态可并行的基础约束。

#### 2.2.5 六阶段工业级工作流深度剖析

传统的 AI 漫画生成多停留在“玩家级”的**提示词抽卡阶段，缺乏长篇连载的能力**。SparkScroll 参考现代动画工业管线，将创作过程拆为六个阶段，并在 Gateway 中实现为可恢复的 episode-by-episode workflow。这种多智能体协同体系在 DGX Spark 的强大算力支撑下，展现出了极高的工程严谨性。

**（1）阶段一：多源文本摄入与生命周期监控（Monitor System）**

用户将百万字级别的古典名著（如《西游记》）原始文本导入系统。Gateway 负责执行切分、项目初始化和 Redis 运行态登记。此时，处于全局制高点的 **Monitor System（监控调度器）** 开始介入。它不仅负责在复杂的异步任务池中分配算力优先级，更通过 FastAPI 接口向前端持续推送实时的系统状态事件流（例如：“第二回数据加载完毕”、“原典编导正在提取人物线”）。这种透明化的监控机制极大地缓解了在执行长达数十分钟的大型处理任务时用户的等待焦虑 。

**（2）阶段二：超长文本降维与提纯（Director Agent）**

在面对古典文学时，AI 往往会被其中庞杂的诗词歌赋、冗余的环境描写和交错的支线情节所淹没。为此，**Director Agent（导演Agent）** 基于 Qwen3.5-9B 启动了长文本压缩工作，项目实测采用128k 上下文，它已经足以在不丢失全局视野的前提下，执行“降维压缩”操作。它像一位经验丰富的总导演，剥离对视觉叙事无用的文本，提取出紧凑、连贯且视觉信息密集的“核心剧本”，并列出该章节出场的关键人物清单。

**（3）阶段三：基于物理版式的分镜裂解（Writer Agent）**

提炼出的核心剧本被无缝传递给 **Writer Agent（分镜编剧Agent，简称"编剧Agent"）**。Writer Agent必须将自然语言剧本转化为能够直接指导作画的视觉分镜。为了符合传统连环画的阅读节奏，它被施加了硬性的模板规则：每一页被限制为 4 到 6 个场景镜头，整个故事单元的生成上限为 20 页。 其最关键的输出是一个符合严格定义的 JSON 数据结构，该协议不仅包含了环境的光影描述、人物的细微动作，还包括了后续文字所需的**预估二维物理坐标**，这是确保后续排版不遮挡重要画面的关键一步。

**（4）阶段四：角色一致性视觉锚定（Character Designer）**

跨越多页保持角色容貌、服饰和气质的一致性，是目前 AI 连环画创作中的"核心难题"。SparkScroll通过 **Character Designer（立绘/角色设计Agent，简称"立绘Agent"）** 引入了工业级的解法。在正片绘制之前，它通过调用庞大的 Qwen-Image-Edit-2511 模型，基于剧本预先生成核心角色的“三视图（正面、侧面、背面）”以及关键道具的标准参考图（Reference Sheets）。这些参考图将作为强力的视觉锚点，永久驻留在扩散模型的条件注入通道中。

**（5）阶段五：高保真底图绘制与空间感知（Drafting & Painting Agent）**

出稿阶段由 **Drafting & Painting Agent（底图出稿Agent，简称"出稿Agent"）** 全面接管了 DGX Spark 的视觉渲染链路。得益于 Qwen-Image-Edit-2511 卓越的几何推理能力和 MMDiT 架构对空间构图的精确控制，该Agent实际采用 Diffusers 与 vLLM-Omni 双驱动：Diffusers 负责稳定的本地推理，vLLM-Omni 兼容链路则用于更快的图像编辑服务调用，把编剧Agent提供的分镜 Prompt 与立绘Agent提供的参考图融合，生成具备构图留白和角色一致性的页面底图。

**（6）阶段六：规避幻觉的程序化图文渲染（Editor Agent & Assembler Agent）**

最终阶段完全摒弃了生成式 AI 固有的幻觉与不可靠性。**Editor Agent（剪辑排版Agent，简称"剪辑Agent"）** 基于 Python 和 PIL（Pillow）图形处理库运行。它精准读取 JSON 文件中的中文字符与留白坐标，直接调用系统底层的高清矢量字库（如典雅的楷体或醒目的黑体），将对白和旁白“物理印刷”到底图的指定区域上。这种降维打击彻底消除了 AI 图片中常见的汉字扭曲现象。 最后，**Assembler Agent（后期合成Agent，简称"合成Agent"）** 采用了一种创新的“流式单页交付（Streaming Page Presentation）”策略 。系统无需挂起等待所有场景渲染完毕；一旦包含 4-6 个分镜的单页处理完成，立即允许前端通过接口获取信息并展示。这种所见即所得的极速体验，是 DGX Spark 极速吞吐能力的最佳证明。*（注：此阶段仅为构思，尚未实现）*



## 3. 技术架构

### 3.1 技术创新点

**3.1.1 重型双大模型常驻显存架构** 

采用“文本主脑（Qwen3.5-9B）+ 视觉渲染中枢（Qwen-Image-Edit-2511）”的双大模型架构，总显存占用约 115GB，实现几万字名著“一口气”读入并保持全局剧情连贯，保持长文本理解与图像生成链路稳定运行。

**3.1.2 多 Agent 协作流水线** 

设计 6 阶段 Agent 工作流，高度还原人类工业化出版流程，每个 Agent 专注于特定任务，通过标准化 JSON 协议流转，实现高效协作。基于 gpt-5.4 规划能力、自研 Python Agent 调用逻辑和调度闭环，完成 Director（导演） -> Writer（编剧） -> CharacterDesigner（角色设计、立绘） -> Drafting（分镜设计、出稿） -> Editor（剪辑） -> Assembler（后期合成） 六阶段串联。

**3.1.3 角色一致性解决方案** 

通过立绘 Agent 预先生成角色参考图，后续所有分镜均以此为基准进行风格对齐，解决了 AI 绘图中角色不连贯的行业难题。

**3.1.4 程序化后期配字** 

采用 Python/PIL 图像处理库进行纯代码配字，拒绝 AI 生字乱码，确保文字渲染的准确性和美观性。（*暂未实现*）

**3.1.5 流式呈现技术** 

前端采用流式按页呈现策略，极大优化用户体验，无需等待全部内容生成即可开始翻阅。

**3.1.6 性能优化策略**

采用内存优化技术，通过调整`gpu-memory-utilization`参数，在保证性能的同时降低内存使用，使模型能够在有限资源环境下运行。文本模型由最新 vLLM v0.19.0 驱动；图像侧组合 Diffusers 与 vLLM-Omni，在稳定性与速度之间做了工程化平衡。

**3.1.7 API标准化**

 设计了标准化的API接口，支持不同模型的统一调用方式，提高系统的可扩展性和可维护性。允许用户根据自己的业务定义 UI 界面与 API 服务端对接。



### 3.2 平台适配性与不可替代性 (Platform Exclusivity)

本项目专为 **NVIDIA DGX Spark (GB10 128G 共享显存)** 量身定制。为彻底拉开与普通消费级显卡（如 RTX 4090 24G）的差距，SparkScroll 基于 GB10 的 128GB 海量统一内存，采用了 **“重型双大模型常驻显存” (Memory-Resident Multi-Model)** 架构，以实现零 Swap 的秒级管线切换。在工业级漫画生成的实际应用中，系统必须同时兼顾复杂的文学逻辑推理与高分辨率的视觉资产生成，这就要求要求一个长上下文语言模型与本地图像编辑模型同时在线。SparkScroll 对 128GB 内存进行了极限预算分配，总占用稳定在 110-115GB ：

| 模块               | 选型                                                        | 显存常驻预算         | DGX Spark的绝对必要性                                        |
| :----------------- | :---------------------------------------------------------- | :------------------- | :----------------------------------------------------------- |
| **文本与逻辑主脑** | Qwen3.5-9B (实际 128K 上下文，vLLM v0.19.0 驱动)            | **~40GB**            | 128K 上下文已足以覆盖绝大多数小说全文理解，稳定支撑剧情压缩、跨集规划与结构化分镜生成。 |
| **视觉渲染中枢**   | Qwen-Image-Edit-2511 / FireRed-Image-Edit-1.1（本地双模型） | **50~60GB**          | 兼顾高保真角色一致性编辑与中文字幕渲染能力，需要为本地图片编辑模型预留足够显存。强行量化会导致排版几何推理能力丧失。 |
| **框架与并发缓冲** | vLLM v0.19.0、FastAPI、Diffusers、vLLM-Omni 适配层、OS      | **~15GB**            | 支撑 API 流转、自研调度、模型服务与图像编辑调用，需要为缓存、调度与系统运行预留安全冗余。 |
| **总计**           | --                                                          | **110~115GB / 128G** | **普通单卡系统完全无法运行该并发管线。** 只有 DGX Spark能实现"零 Swap"的极致秒级响应，支撑多 Agent 极速切换与流式呈现。 |

这种高达 110-115GB 的超高密度管线驻留，使得SparkScroll在执行复杂的六阶段工序时，各智能体能够像齿轮一样紧密咬合并快速响应 。在非统一内存或任何低于 128GB 内存上限的设备上是物理层面无法企及的，从而完美证明了SparkScroll项目对 DGX Spark 平台的绝对刚需。

### 3.4 整体分层架构设计

系统采用分层架构，确保“接口、调度、资源、状态、模型、安全、观测”彼此解耦：

1. **表现层 (API Layer)**  
   基于 **FastAPI** 提供 RESTful 接口，负责项目创建、参数校验、文本接入、语义保护切分、任务启动/停止、进度查询、页面预览和提示词管理。

2. **调度层 (Orchestration Layer)**  
   基于自研 Python 异步工作流协调器、IndependentAgentRunner 与 Redis 运行态仓储组织多 Agent 工作流，不依赖 LangGraph；并借助 gpt-5.4 的规划能力优化任务拆解、执行状态流转、失败重试、同步点控制和队列投递。

3. **资源管理层 (Resource Management Layer)**  
   统一管理本地文件系统资源，负责系统级公共预设、项目级私有资源、生成图片资产及资源隔离校验。

4. **运行态数据与记忆层 (Runtime State & Memory Layer)**  
   基于 **Redis** 管理项目元数据、队列、锁、运行状态、单集内容、分镜结果、导演/编剧记忆、停止信号等高频动态数据。

5. **模型适配层 (Model Gateway)**  
   基于 vLLM v0.19.0、Diffusers、本地推理服务和兼容云端 API 的适配器层，负责逻辑模型与视觉模型的统一调用、超时控制、错误重试与本地/云端切换。

6. **安全执行层 (Security & Sandbox Layer)**  
   当前基于 API 鉴权、配置约束、项目级资源隔离和运行时边界控制来保护文件访问、密钥读取与模型调用；后续再补充更细粒度的沙箱策略。

7. **观测与诊断层 (Observability Layer)**  
   基于 Python `logging` 与结构化日志规范，覆盖系统运行日志、项目运行日志、模型调用诊断和 API 审计链路。

   

### 3.5 资源管理层设计

资源管理层负责管理**本地文件系统中的低频、可持久化、可复用资产**，并严格执行项目边界控制。

#### 3.5.1 访问隔离原则

*   **严格按 Project ID 隔离**：除系统级公共预设外，所有项目资产均必须在对应项目 ID 命名空间下读写。
*   **禁止跨项目读取资源信息**：任何 Agent 不允许读取其他项目的原始文本、角色资产、分镜文件、底图和页面文件。
*   **系统公共预设只读访问**：通用的连环画构图框架、排版样式等系统预设可被所有项目读取，但必须保持只读，不允许被项目运行过程覆盖。
*   **停止任务不删除已产物**：调用停止接口只中断后续执行，不删除已经落盘的文本切片、角色资源、分镜结果、底图和页面资源。

#### 3.5.2 资源分类

1. **系统级公共预设**  
   包含通用项目预设的连环画构图框架、排版样式、页面布局规则、文字框位置规则等公共模板资产。此类资源由系统维护，全项目共享，只读访问。

2. **项目文本资源**  
   包含原始小说文本及经 API 层语义保护切分后的 `.txt` / `.md` 片段。切分结果是导演 Agent 的直接输入来源。

3. **项目模版资源**  
   包含该项目使用的构图模版、页面排版模版、镜头样式规则和页面渲染配置，主要以 `.json` 为核心描述格式。

4. **角色资源库**  
   仅包含两类资产：

   *   **角色图片**：`.jpg`、`.png`
   *   **角色配置**：`.json`

   角色资源库不存放 Markdown 文稿或额外文本说明，角色语义信息统一通过 JSON 配置承载，角色视觉参考统一通过图片承载。

5. **图像与页面资产**  
   包含立绘结果图、出稿底图、中间编辑图、最终渲染页面及页面聚合清单等资产。



### 3.6 运行态数据与 Redis 管理设计

Redis 负责承载**高频读写、任务调度、状态追踪和运行态记忆**，是系统的动态控制中枢。

| 数据类别   | 推荐 Key 示例                           | 说明                                                         |
| :--------- | :-------------------------------------- | :----------------------------------------------------------- |
| 项目元数据 | `project:{id}:meta`                     | 存储项目名称、模型选择、创建时间、分片摘要等基础信息。       |
| 项目状态   | `project:{id}:status`                   | 存储项目状态机，如 `CREATED / READY / RUNNING / STOPPED / COMPLETED / FAILED`。 |
| 集列表     | `project:{id}:episodes`                 | 存储项目下已规划集的顺序索引。                               |
| 单集内容   | `project:{id}:episode:{ep}:outline`     | 存储导演生成的单集纲要、角色新增摘要、剧情目标。             |
| 分镜内容   | `project:{id}:episode:{ep}:storyboards` | 存储编剧输出的镜头 JSON 列表。                               |
| 导演记忆   | `project:{id}:memory:plot`              | 存储宏观剧情梗概、人物关系、长期伏笔。                       |
| 编剧记忆   | `project:{id}:memory:scene`             | 存储最近若干镜头状态，确保镜头衔接一致。                     |
| 任务队列   | `queue:text` / `queue:image`            | 文本任务与图像任务分队列排队执行。                           |
| 资源锁     | `lock:model:text` / `lock:model:image`  | 控制文本模型与图像模型各自仅 1 并发。                        |
| 停止信号   | `project:{id}:control:stop`             | 标记项目已发出停止请求，Worker 轮询并安全退出。              |
| 任务运行态 | `project:{id}:task:{task_id}`           | 存储单任务状态、重试次数、耗时、错误信息。                   |

Redis 中保存的是**运行态数据**而非长期冷存档；长期可复用的页面与图像结果应落盘到资源管理层，避免 Redis 承担大体积二进制资源存储。



### 3.7 核心数据对象与状态机 (Domain Model & State Machine)

#### 3.7.1 核心数据对象

系统建议统一以下核心对象模型：

1.  **Project**  
    表示一次完整的连环画生成任务，包含模型配置、切片信息、资源根标识、状态机和统计信息。
2.  **Episode**  
    表示导演规划出的单集单元，是导演、编剧、立绘、出稿协同的最小业务单元之一。每一集对应一份纲要、一组角色增量和一组分镜结果。
3.  **Scene / Shot**  
    表示编剧输出的单个镜头，是后续生图、排版和页面合成的直接依据。推荐以 JSON 对象形式描述构图、人物、动作、对白和镜头说明。
4.  **Character Asset**  
    表示角色资源库中的单角色资产集合，包括角色主图、补充参考图和角色配置 JSON。
5.  **Page**  
    表示最终输出的单页连环画结果，包含页面编号、底图、渲染图、文本覆盖状态和预览地址。

#### 3.7.2 项目状态机

项目状态机按如下约束实现：

| 状态        | 含义                                         | 允许的下一状态                      |
| :---------- | :------------------------------------------- | :---------------------------------- |
| `CREATED`   | 项目已创建，但尚未完成切片初始化             | `READY` / `FAILED`                  |
| `READY`     | 切片与资源目录已就绪，可启动                 | `RUNNING` / `FAILED`                |
| `RUNNING`   | 正在执行多 Agent 流水线                      | `STOPPING` / `COMPLETED` / `FAILED` |
| `STOPPING`  | 已接收停止指令，等待 Worker 安全退出         | `STOPPED` / `FAILED`                |
| `STOPPED`   | 已停止，但已完成产物保留，可重新启动继续执行 | `RUNNING` / `FAILED`                |
| `COMPLETED` | 全流程完成                                   | --                                  |
| `FAILED`    | 出现不可恢复错误                             |                                     |

#### 3.7.3 任务状态机

所有 Agent 任务统一复用以下任务状态：
`PENDING -> QUEUED -> RUNNING -> SUCCEEDED / FAILED / CANCELLED`

其中：

*   `CANCELLED` 表示任务因停止信号或人工中止而退出。
*   已经完成的任务不会因为项目进入 `STOPPED` 而被回滚。
*   `start` 在 `STOPPED` 状态下应表现为**恢复执行**，而不是重新清空已有内容。



### 3.8 核心多 Agent 协作流水线 (Pipeline Architecture)

SparkScroll 摒弃了简单的端到端黑盒，设计了高度还原人类工业化出版流程的 **6 阶段 Agent 工作流**：

#### 3.8.1 原始输入与分段 (User Input)

* **动作**：用户输入名著或小说原文。
* **处理**：支持通过行号或正则截断表达式进行初步的章节分段。

#### 3.8.2 进度管理中枢 (Monitor System)

* **动作**：贯穿全生命周期的系统级调度器。
* **处理**：由自研 Workflow Coordinator 管理项目状态、任务切换与停止恢复，并实时向前端汇报工作流进展，提供清晰的进度感知。

#### 3.8.3 导演 Agent (Director - 文本驱动)

* **动作**：Qwen3.5-9B（实际 128K 上下文，vLLM v0.19.0 驱动）。
* **处理**：对原始长文本进行“降维压缩”。剔除冗余支线，精简核心角色与关键场景，提炼出紧凑的主线剧本。

#### 3.8.4 编剧 Agent (Writer - 文本驱动)

* **动作**：Qwen3.5-9B（模板化结构输出）。
* **处理**：将导演的剧本进行视觉化“镜头拆分”。
* **规则限制**：严格通过 Prompt 模板控制节奏，规划**每页 4-6 个场景**。整部连环画篇幅控制在 20 页以内，总计不超过 120 个场景。输出标准化 JSON 剧本（含场景描述、留白坐标预估、对白）。

#### 3.8.5 立绘 Agent (Prop/Character Designer - 图像驱动)

* **引擎**：Qwen-Image-Edit-2511 / FireRed-Image-Edit-1.1（Diffusers 稳定驱动）。
* **任务**：在正式绘制分镜前，根据导演设定的角色名单，预先绘制“核心角色三视图”与“关键物件图”（Reference Sheets）。
* **意义**：这是解决 AI 绘图“角色不连贯”的杀手锏。后续所有分镜均以此为基准进行风格对齐。

#### 3.8.6 出稿 Agent (Drafting & Painting - 图像驱动)

* **引擎**：Diffusers + vLLM-Omni 双驱动。
* **任务**：接收编剧的 JSON 分镜 Prompt 与立绘 Agent 的角色参考图，兼顾稳定性与出图速度。在生成图像时，根据 JSON 预留“构图留白”，合并输出每页的基础图像。

#### 3.8.7 剪辑Agent (Edit Agent，开发中)

* **引擎**：Python / PIL (Pillow) 图像处理库。
* **任务**：**拒绝 AI 生字乱码**。通过纯代码读取编剧 JSON 中的对白文本与坐标，加载中文字体（如黑体/楷体）模板，将文字完美渲染至出稿 Agent 预留的空白区域（对话框或旁白栏）。

#### 3.8.8 合成Agent (Assembler Agent，开发中)

* **机制**：前端采用“流式按页呈现”策略。不需要等待全部 120 个场景跑完，**每生成完一页（4-6 镜 + 当前页面定稿），即刻推送到前端供用户翻阅**，极大优化用户体验。
* **扩展**：理想状态下，最终生成连贯的 PDF 画册（初期版本设为 Optional 目标）。



### 3.9 技术栈及选型依据

#### 3.9.1 Agent技术栈选型说明

**（1）vLLM v0.19.0（专用推理加速）**

vLLM v0.19.0通过PagedAttention、共享内存优化、连续批处理、智能调度等核心技术，实现了显存利用率与GPU计算效率的双重优化，显著提升了推理吞吐量并降低了延迟

- **PagedAttention机制**：vLLM v0.19.0借鉴了操作系统虚拟内存的分页技术，将GPU显存利用率从传统方法的约40%提升至90%以上。该机制将固定的键值缓存（KV Cache）切分为固定大小的块（如每块包含16个Token），并根据请求的实际需求动态分配这些块。通过块表记录逻辑与物理块的映射关系，实现了几乎零浪费的内存管理，从而显著提高了显存利用率。
- **共享内存优化**：当多个请求拥有相同前缀时，vLLM v0.19.0允许它们共享同一份KV Cache，从而节省高达55%的内存。这一特性在对话系统等场景中尤为重要，因为对话历史往往包含大量重复信息。

- **连续批处理（Continuous Batching）**：vLLM v0.19.0改变了传统批处理必须等待整批任务完成的模式，允许新请求随时加入。每次GPU完成一次前向传播后，调度器会检查哪些请求已经准备好，并将它们重新打包成一个新的批次继续处理。这种动态批处理方式显著提高了GPU的利用率和处理效率，吞吐量可提升2-4倍。
- **智能调度算法**：vLLM v0.19.0根据请求的长度和优先级动态调整资源分配，确保GPU资源得到充分利用。例如，对于长请求和短请求，调度器会合理分配计算资源，避免短请求被长请求阻塞。

**（2）FastAPI 与 自研 Agent 调度（高速异步并发）**

整个多智能体系统的神经中枢建立在 FastAPI 构建的高性能异步接口之上，这使得系统能够以非阻塞的方式同时处理大量图文生成请求。

**（3）Diffusers + vLLM-Omni 双图像驱动（多模态视觉桥接）**

视觉任务采用双驱动策略：Diffusers 用于稳定的本地图片编辑推理，vLLM-Omni 用于更快的图像编辑服务调用。当前本地主要接入 Qwen-Image-Edit-2511 与 FireRed-Image-Edit-1.1 两种模型，其中前者更适合高保真角色一致性编辑，后者在中文文本渲染场景中实测更优。

**（4）Python PIL（降维打击的物理排版引擎：策划中）** 

当前最前沿的扩散模型在生成复杂的中文字体时，依然面临着严峻的“鬼画符”和乱码问题 。SparkScroll拟采用了“降维打击”的工程思路，直接剥夺了 AI 生成文字的权限。通过 Python 的 PIL（Pillow）物理渲染排版库，系统读取 JSON 中由大语言模型生成的纯净对话文本以及空间坐标，调用系统底层的矢量字库（如标准黑体、正楷），将文字以完美的印刷级清晰度渲染到 AI 预留的画面空白处。

（*BTW：由于大赛时间有限，第四部分内容仅为技术思路，尚未实现*） 



#### 3.9.2 前端技术栈选型说明

**（1）Web UI技术概述**

尽管常见的Web UI技术包含：JavaScript、jQuery、Vue.js 、React.js、Angular.js等，但是在数据应用领域，特别是人工智能领域，Streamlit和Gradio已经成为了构建数据应用程序WebUI的主流框架。

**Streamlit** 是一种用于构建数据应用程序的 Python 库：

- Streamlit 主要用于构建数据科学和机器学习应用程序，通过简单的 Python 脚本即可实现交互式的数据可视化和分析。
- Streamlit 提供了快速迭代、简单易用的特性，使得数据科学家和分析师可以更快地创建漂亮的数据应用程序，而无需深入学习前端开发技术。
- 与 JavaScript 技术相比，Streamlit 在构建数据应用程序方面更为方便，但在灵活性和定制化方面可能会有所不足。

**Gradio** 是一个用于快速构建机器学习界面的 Python 库。Gradio 也旨在简化机器学习模型的部署和展示，但它专注于提供一个简单易用的界面来展示模型的输入和输出，而不涉及其他数据分析或可视化方面的功能。

**（2）Streamlit 选型依据**

Gradio 和 Streamlit 之间的主要区别包括：

1. 重点功能：
   1. Gradio 主要用于构建机器学习模型的交互式界面，使用户能够直观地了解模型的输入和输出，并与模型进行交互。
   2. Streamlit 则更加通用，可以用于构建各种类型的数据应用程序，包括数据分析、可视化、文本处理等，不仅局限于机器学习模型的展示。
2. 界面设计：
   1. Gradio 提供了一系列预定义的界面组件，如输入框、滑块、下拉框等，使用户可以轻松地构建简单且美观的界面。
   2. Streamlit 也提供了类似的界面组件，但其更加灵活，用户可以通过代码自定义界面的外观和交互方式，从而实现更多样化的应用程序设计。
3. 学习曲线：
   1. Gradio 的学习曲线相对较低，适合那些想要快速构建机器学习模型展示界面的用户，尤其是对前端开发不太熟悉的人员。
   2. Streamlit 的学习曲线可能稍高一些，但提供了更多的灵活性和定制化选项，适用于更广泛范围的数据应用场景。

考虑到以上特点，加上本项目组成员已掌握的相关框架现状，项目组决定选择streamlit作为本项目的Web UI框架。

**（3）Streamlit工作流程**

Streamlit 的工作流程如下：

- 每次用户交互均需要从头运行全部脚本。
- Streamlit 根据 widget 状态为每个变量分配最新值。
- 缓存保证 Streamlit 重用数据和计算。



## 4. 使用的NVIDIA工具、平台、SDK与开源模型

### 4.1 硬件适配

本项目专为 NVIDIA DGX Spark (128G 共享显存)平台量身定制，**DGX Spark** 是基于 **NVIDIA Grace Blackwell (GB10)** 架构的桌面级 AI 超级计算机。其核心特性直接决定了软件栈的性能表现：

*   **计算核心**：搭载 Blackwell 架构 GPU，支持 **第 5 代 Tensor Core**，提供高达 **1 PFLOPS** 的 **FP4** AI 性能。
*   **存储架构**：128GB LPDDR5x **统一内存 (Unified Memory)**，CPU 与 GPU 之间实现高速连接。
*   **计算单元**：20 核 Arm CPU (Cortex-X925/A725)，专为高效调度 AI 工作负载设计。

### 4.2 Nvidia工具使用

本项目中使用到了Nvidia的以下相关工具和平台：

- **TensorRT-LLM**： 这是 NVIDIA 开发的开源推理加速库，专为优化大型语言模型（LLM）在 NVIDIA GPU 上的推理性能而设计。TensorRT-LLM 通过深度图级优化、量化技术和并行机制，显著提升了推理速度并降低了延迟。我们使用TensorRT-LLM来跑Nemotron-3-Nano-30B-A3B-NVFP4模型，该技术有以下特点：
  - **模型优化**：TensorRT-LLM 采用层级融合、内核选择和精度调整等技术，对模型进行深度优化。它支持 FMHA（Fused Multi-Head Attention）和 MMHA（Masked Multi-Head Attention）等优化内核，有效减少了计算冗余，降低了内存占用。
  - **量化技术**：为了进一步降低推理延迟和显存占用，TensorRT-LLM 支持多种量化选项，如 FP16、INT8、FP8 等。用户可以根据具体需求选择合适的量化级别，实现性能与精度的平衡。例如，INT8 量化可以将模型大小减半，同时保持较高的推理精度。
  - **内存管理**：TensorRT-LLM 通过智能内存分配和分页注意力机制，优化了内存使用。它支持 Paged KV Cache 技术，将 KV Cache 分页存储，从而支持处理超出 GPU 内存容量的模型，提高了内存的利用率。
  - **并行处理与动态批处理**：TensorRT-LLM 支持多线程并行处理，充分利用 NVIDIA GPU 的计算能力，加速模型推理。此外，它还支持动态批处理（In-flight Batching），通过同时处理多个请求来优化文本生成，减少等待时间并提高 GPU 利用率。
  - **易用性与扩展性**：TensorRT-LLM 提供了简洁易用的 Python API，允许用户定义大型语言模型并构建包含先进优化的 TensorRT 引擎。该 API 设计类似于 PyTorch，使得具有 PyTorch 经验的开发者能够轻松迁移和集成。同时，TensorRT-LLM 支持多种预定义模型，用户可以基于这些模型进行扩展或定制，满足不同场景下的推理需求。
- **vLLM-Omni**：作为图像编辑的快速调用链路，vLLM-Omni 显著提升了本地图片编辑服务的响应速度。它优化了图像编辑任务的执行流程，减少了不必要的计算开销，使得用户能够更快地获得编辑结果，提升了用户体验。
- **vLLM v0.19.0**：在实际部署中，我们选用了 vLLM v0.19.0 来运行 Qwen3.5-9B 大语言模型。该工具通过 OpenAI-compatible 接口接入 Gateway，稳定支撑了 128K 上下文的处理。vLLM 的高效性和稳定性确保了语言模型在复杂场景下的可靠运行，为项目提供了强大的语言处理能力。
- **NVIDIA CUDA**：作为 GPU 加速的基石，我们选用了 **CUDA 12.9/13.0** 版本。该版本能够充分发挥 NVIDIA GPU 的并行计算能力，为深度学习模型的训练和推理提供强大的硬件支持，确保计算任务的高效执行。
- **PyTorch**：我们采用 PyTorch 2.10.0 作为深度学习框架，它结合了 CUDA 的 GPU 加速能力，实现了模型推理的高效性。PyTorch 的灵活性和易用性使得我们能够快速构建和训练复杂的深度学习模型，满足项目多样化的需求。
- **HuggingFace Diffusers** ：该工具原生驱动了 Qwen-Image-Edit-2511 和 FireRed-Image-Edit-1.1 等图像编辑模型。它能够灵活地注入 Image/Prompt 条件，为图像编辑任务提供了强大的支持。通过 HuggingFace Diffusers，我们能够轻松实现图像的创意编辑和风格转换。

### 4.3 Nvidia开源模型使用

- **Nemotron-3-Nano-30B-A3B-NVFP4**：该模型是Nvidia推出的模型，采用 **NVFP4** 格式，专门为 Blackwell 优化，在 Blackwell 架构上，其吞吐量可达同类模型的 **3倍 以上**。其架构为**MoE + Mamba + Transformer**，其 31.6B 的总参数量能提供极强的知识储备，而 3.6B 的激活参数确保了极速响应。

### 4.4 框架对比：TensorRT-LLM vs vLLM

| 维度               | TensorRT-LLM (推荐)                                          | vLLM (通用)                                                  |
| :----------------- | :----------------------------------------------------------- | :----------------------------------------------------------- |
| **Blackwell 适配** | **原生深度优化**。第一时间支持 GB10 的 FP4 精度和第 5 代 Tensor Core，能跑满 1 PFLOPS 性能。 | **社区适配中**。虽然支持 PagedAttention，但对 Blackwell 特有的 FP4 硬件加速支持通常滞后。 |
| **统一内存利用**   | 针对 Grace Blackwell 的 **内存相干性** 进行了底层优化，在处理超大上下文或多模型切换时延迟更低。 | 采用标准 CUDA 内存模型，虽然在 Blackwell 上能跑，但无法完全压榨 Arm CPU 与 GPU 之间的带宽优势。 |
| **NVFP4 算子**     | 内置针对 NVFP4 格式的专用 Kernel，是实现 Blackwell 宣称性能的关键。 | 目前仍以 FP16/BF16 和 FP8 为主，FP4 支持尚在开发或实验阶段。 |
| **量化感知**       | 支持使用 NVIDIA 官方量化工具（Model Optimizer）进行量化感知训练/蒸馏。 | 侧重于通用量化算法（如 AWQ/GPTQ），对 Blackwell 专属的 FP4 硬件格式支持度有限。 |

由上可知，在 DGX Spark (Blackwell 架构) 上，TensorRT-LLM针对vLLM的优势进一步拉大。

### 4.5 模型对比：Nemotron-3-Nano (NVFP4) vs Qwen3.5-9B

| 维度           | Nemotron-3-Nano-30B-A3B-NVFP4                                | Qwen3.5-9B (Dense)                                           |
| :------------- | :----------------------------------------------------------- | :----------------------------------------------------------- |
| **精度契合度** | **完美契合**。该模型专门为 Blackwell 设计，采用 **NVFP4** 格式，能在 DGX Spark 上实现极致吞吐。 | **通用精度**。通常运行在 BF16 或 FP8 精度。在相同硬件上，其推理密度低于 NVFP4 格式。 |
| **架构优势**   | **MoE + Mamba + Transformer**。在 DGX Spark 的 128GB 内存中，其 31.6B 的总参数量能提供极强的知识储备，而 3.6B 的激活参数确保了极速响应。 | **Dense Transformer**。架构传统且稳健，逻辑推理能力强，但在 9B 级别，其知识密度和吞吐潜力受限于稠密计算。 |
| **吞吐效率**   | 在 Blackwell 架构上，其吞吐量可达同类模型的 **3x 以上**。    | 表现稳定，但在 DGX Spark 这种具备 FP4 硬件加速的设备上，性价比（TPS/Watt）不如优化后的 Nemotron。 |
| **最佳用途**   | 适合作为 **Agent 核心节点**，处理高频、高并发的指令遵循和信息处理。 | 适合处理**复杂长文本推理**或作为高精度逻辑基座模型。         |

## 4.6 落地选型建议

DGX Spark 的核心价值在于 **FP4 精度** 和 **Grace Blackwell 统一内存**。

*   **TensorRT-LLM** 是目前唯一能完全解锁 GB10 硬件潜力的框架。
*   **Nemotron-3-Nano (NVFP4)** 是目前最能代表该硬件“性能巅峰”的展示模型，它在 4-bit 精度下依然保持了极高的模型能力。

建议按照以下方式进行落地选型：

*   **开发/实验阶段**：建议部署 **vLLM + Qwen3.5-9B**。利用 vLLM 的易用性快速验证业务逻辑，利用 Qwen3.5 优秀的中文理解和逻辑能力构建 Prompt 模板。
*   **高性能生产/Agent 部署**：切换至 **TensorRT-LLM + Nemotron-3-Nano (NVFP4)**。
    1.  使用 TensorRT-LLM 编译针对 GB10 的专用 Engine。
    2.  利用 DGX Spark 预装的 **NVIDIA AI Enterprise** 软件栈进行监控。
    3.  在同等功耗下，该组合可提供 **2x-4x 的并发处理能力**，显著降低单次推理成本。

## 6. 未来展望

### 6.1 短期规划

短期将并行推进两项工作：一是继续开发剪辑Agent和合成Agent，但不把它作为当前演示版本的阻塞项；二是开展更适合画漫画场景的模型二次训练，重点增强同一画面上不同镜头之间的时空关系理解与分镜连续性。

### 6.2 长期规划

1. **模型专项训练与扩展**：在现有 Qwen-Image-Edit-2511 / FireRed-Image-Edit-1.1 经验基础上，继续探索更适合漫画生产的模型，并逐步扩展到视频生成、3D 内容创作等方向

2. **性能优化**：进一步优化模型推理性能，探索模型量化、剪枝等技术，在保证生成质量的同时降低资源需求

3. **用户界面改进**：开发更直观、更易用的用户界面，支持拖拽式操作和实时预览，提升用户体验

4. **云服务部署**：除了本地部署外，计划提供云服务版本，让用户可以通过互联网访问平台，无需本地配置

5. **行业应用拓展**：针对不同行业的特定需求，开发行业专用的模型和工具，如广告设计、教育内容创作、游戏资产生成等

6. **开源社区建设**：将项目开源，建立社区生态，吸引更多开发者参与，共同推动平台的发展和创新

通过以上规划，我们相信 SparkScroll 平台将成为内容创作领域的重要工具，为用户提供更加智能、高效的内容生成和编辑体验。



## 7. 结语

星火绘卷（SparkScroll）已经向DGX Spark 全栈 AI 开发黑客松大赛交出了一份堪称完美的工业级答卷。它不仅如手术刀般精准地切中了 GB10 超级芯片在 128G 统一大内存和极速多模型流转协同上的底层架构优势，更以一种令人惊艳的工程执行力，将深邃的古典文学遗产与最前沿的生成式人工智能进行了一次充满浪漫主义色彩的融合。

这不仅是一次在极客赛场上的算力破局，更是一次横跨时代的文化接力。在 DGX Spark 汹涌澎湃的硅基算力赋能下，那些曾经沉睡在泛黄纸页间的千古篇章与英雄史诗，正被重塑为一帧帧气韵生动、跃然屏上的数字画卷，以前所未有的姿态，浩浩荡荡地再次卷入现代数字文明的洪流之中。
